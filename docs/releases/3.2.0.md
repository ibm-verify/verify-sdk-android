# IBM Verify SDK for Android - Release 3.2.0

## Overview

Version 3.2.0 is a significant enhancement release focused on improving reliability, performance, and developer experience. This release includes **critical fixes** for token persistence, comprehensive performance optimizations, and enhanced thread safety throughout the SDK.

## 🚨 Breaking Changes

### 1. MFA SDK - Service Constructor Signature Changes

**Impact: HIGH**

The `CloudAuthenticatorService` and `OnPremiseAuthenticatorService` constructors now require `HttpClient` parameter and accept optional (but **strongly recommended**) `TokenPersistenceCallback` parameter.

#### Changes:

**Before (3.1.x):**
```kotlin
val service = CloudAuthenticatorService(
    _accessToken = token.accessToken,
    _refreshUri = refreshUri,
    _transactionUri = transactionUri,
    _authenticatorId = authenticatorId
)

// HttpClient passed to methods
service.refreshToken(refreshToken, additionalData, httpClient)
```

**After (3.2.0):**
```kotlin
val service = CloudAuthenticatorService(
    _accessToken = token.accessToken,
    _refreshUri = refreshUri,
    _transactionUri = transactionUri,
    _authenticatorId = authenticatorId,
    httpClient = NetworkHelper.getInstance,
    persistenceCallback = repository  // CRITICAL: Implement this!
)

// HttpClient no longer in method parameters
service.refreshToken(refreshToken, additionalData)
```

#### Migration Steps:

1. **Add `httpClient` parameter** to service constructor
2. **Implement `TokenPersistenceCallback`** (see below - this is CRITICAL)
3. **Remove `httpClient` from method calls**

### 2. MFA SDK - Token Persistence Callback Required

**Impact: CRITICAL**

**YOU MUST IMPLEMENT `TokenPersistenceCallback`** to prevent the "hosed authenticator" scenario.

#### Why This Matters:

Without token persistence, if your app crashes after a token refresh but before saving to the database, the authenticator becomes permanently broken because:
1. Server activates the new token
2. Server invalidates the old token
3. App crashes before saving new token
4. On restart, old token is loaded
5. All API calls fail with 401 Unauthorized

#### Implementation Example:

```kotlin
class AuthenticatorRepository(
    private val database: Database
) : TokenPersistenceCallback {

    override suspend fun onTokenRefreshed(
        authenticatorId: String,
        newToken: TokenInfo
    ): Result<Unit> {
        return try {
            // CRITICAL: Save to database
            val authenticator = database.getAuthenticator(authenticatorId)
            authenticator.token = newToken
            database.updateAuthenticator(authenticator)
            
            Log.d(TAG, "Token persisted successfully")
            Result.success(Unit)
        } catch (e: Exception) {
            Log.e(TAG, "CRITICAL: Token persistence failed", e)
            Result.failure(e)
        }
    }
}

// Use in service creation
val service = CloudAuthenticatorService(
    _accessToken = token.accessToken,
    _refreshUri = refreshUri,
    _transactionUri = transactionUri,
    _authenticatorId = authenticatorId,
    httpClient = NetworkHelper.getInstance,
    persistenceCallback = repository  // Pass your implementation
)
```

### 3. MFA SDK - Service Lifecycle Changes

**Impact: MEDIUM**

Services are now **immutable value objects** that must be recreated when tokens change.

#### Changes:

**Before (3.1.x):**
```kotlin
// Service could be reused after token refresh
val service = createService(oldToken)
service.refreshToken(...)
// Continue using same service instance
```

**After (3.2.0):**
```kotlin
// Service must be recreated after token refresh
val service = createService(oldToken)
service.refreshToken(...).onSuccess { newToken ->
    // Old service is now obsolete - create new one
    val newService = createService(newToken)
    // Use newService for subsequent operations
}
```

#### Benefits:
- Thread-safe by design (no synchronization needed)
- Clear lifecycle semantics
- No hidden state mutations
- Lightweight (fast to create)

### 4. Core SDK - NetworkHelper Thread Safety

**Impact: LOW**

`NetworkHelper.getInstance` is now `@Synchronized` with lazy initialization.

#### Changes:

- `client` and `defaultClient` properties are now `@Volatile`
- `getInstance` uses synchronized lazy initialization
- `invalidateClient()` replaces `reinitializeClient()`

This change is **backward compatible** but improves thread safety.

## ✨ New Features

### 1. Critical Token Persistence Fix

**CRITICAL FIX** that prevents authenticators from becoming permanently unusable.

#### What It Fixes:

The SDK now ensures that token refresh operations are **atomic**:
- Token is persisted to database **before** refresh returns success
- If persistence fails, entire refresh fails
- Old token remains valid if persistence fails
- Prevents partial state where server has new token but app doesn't

#### Implementation:

```kotlin
// In CloudAuthenticatorService.refreshToken()
persistenceCallback?.let { callback ->
    val persistResult = callback.onTokenRefreshed(authenticatorId, newTokenInfo)
    persistResult.onFailure { error ->
        Log.e(TAG, "CRITICAL: Token persistence failed")
        return Result.failure(
            Exception("Token refresh succeeded but persistence failed: ${error.message}", error)
        )
    }
    Log.d(TAG, "Token persisted successfully")
}
```

### 2. Certificate Transparency Support

Optional Certificate Transparency (CT) verification for enhanced security.

**Resolves:** [#44 - Certificate Transparency Support](https://github.com/ibm-verify/verify-sdk-android/issues/44)

#### Features:

- **Interceptor-based approach** (SDK best practice)
- **Scoped to SDK** (won't conflict with client app)
- **Disabled by default** (backward compatible)
- **Comprehensive documentation**

#### Usage:

```kotlin
import com.appmattus.certificatetransparency.certificateTransparencyInterceptor

NetworkHelper.certificateTransparencyInterceptor = certificateTransparencyInterceptor {
    // Exclude specific domains
    -"*.example.com"
    -"test.com"
    
    // Re-include specific subdomain
    +"allowed.example.com"
}
```

See `docs/CERTIFICATE_TRANSPARENCY_GUIDE.md` for complete documentation.

### 3. Lazy Logging Extensions

New lazy logging extensions prevent string allocation when logging is disabled.

#### Features:

- **Android Log extensions**: `logDebug`, `logInfo`, `logWarn`, `logError`
- **SLF4J Logger extensions**: `entering`, `exiting`, `debugLazy`, etc.
- **Performance improvement** in production builds

#### Usage:

```kotlin
// Instead of:
Log.d(TAG, "Processing transaction ${transaction.id} for user ${user.name}")

// Use:
logDebug(TAG) { "Processing transaction ${transaction.id} for user ${user.name}" }
```

The lambda is only evaluated if the log level is enabled, avoiding unnecessary string allocations.

### 4. Enhanced KeystoreHelper Documentation

Comprehensive documentation with usage examples for:
- AES key management
- Key pair generation (RSA, EC)
- Biometric integration
- Security features

### 5. Type-Safe JSON Parsing

New `@Serializable` data classes for type-safe JSON parsing:

- `TransactionData` - Transaction information
- `AdditionalDataItem` - Transaction metadata

Replaces `org.json` with `kotlinx.serialization` for:
- Compile-time type checking
- Null safety
- Reduced code complexity
- Better error messages

   - Immutable value objects

1. **COSEKey Performance**

   - Shared CBORMapper instances

### Adaptive SDK

1. **Modern Lifecycle APIs**
   - Migrated from deprecated `LifecycleObserver` with `@OnLifecycleEvent` to `DefaultLifecycleObserver`
   - Explicit lifecycle method overrides (`override fun onStart()`) for better type safety
   - Compile-time checking instead of runtime annotation processing
