# IBM Verify SDK for Android - Release 3.2.4

## Overview

Version 3.2.4 is a patch release focused on correctness and resilience for on-premise MFA flows, with targeted improvements to SSL handling, authenticator identifier management, transaction ownership filtering, and token persistence behavior.

This release addresses a class of issues where on-premise deployments use multiple identifiers for the same logical authenticator:

- a client-generated `tenant_id`, used locally by the SDK and application
- a server-generated `authenticator_id`, returned by the server and used in transaction correlation

The main goal of this release is to ensure those identifiers remain distinct and are used consistently in the correct places across registration, token refresh, and transaction processing.

In addition, 3.2.4 introduces controlled SSL bypass support for trusted on-premise environments using self-signed certificates, with explicit application opt-in and authenticator-level activation.

## Breaking Changes

None.

This is a backward-compatible patch release. However, applications integrating with on-premise environments should review the SSL and identifier handling notes in this document to ensure they are using the intended configuration model.

## New Features

### 1. Controlled SSL Bypass for Trusted On-Premise Environments

**New:** [`NetworkHelper.allowInsecureSSL`](../../sdk/core/src/main/java/com/ibm/security/verifysdk/core/helper/NetworkHelper.kt:103) and [`NetworkHelper.createInsecureClient()`](../../sdk/core/src/main/java/com/ibm/security/verifysdk/core/helper/NetworkHelper.kt:323)

#### Overview

Some on-premise IBM Verify Access deployments use self-signed or privately issued certificates that are not trusted by the default Android trust store. Version 3.2.4 adds explicit support for these environments through a controlled SSL bypass mechanism.

This support is intentionally gated behind a two-level model:

1. **Application-level permission** via [`NetworkHelper.allowInsecureSSL`](../../sdk/core/src/main/java/com/ibm/security/verifysdk/core/helper/NetworkHelper.kt:103)
2. **Authenticator-level requirement** via `ignoreSSLCertificate`, derived from QR code options and carried by [`OnPremiseAuthenticator`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/model/onprem/OnPremiseAuthenticator.kt:20)

Both conditions must be satisfied before the SDK uses an insecure HTTP client.

#### Code Changes

- Added [`NetworkHelper.allowInsecureSSL`](../../sdk/core/src/main/java/com/ibm/security/verifysdk/core/helper/NetworkHelper.kt:103) as the global application opt-in flag
- Added [`NetworkHelper.createInsecureClient()`](../../sdk/core/src/main/java/com/ibm/security/verifysdk/core/helper/NetworkHelper.kt:323) to create a dedicated insecure client when explicitly allowed
- Updated [`MFAServiceController.initiate()`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/MFAServiceController.kt:129) to select the appropriate client for on-premise authenticators:
  - secure provided/default client when `ignoreSSLCertificate=false`
  - dedicated insecure client when `ignoreSSLCertificate=true`
- Preserved the SSL requirement on the service instance through [`OnPremiseAuthenticatorService.ignoreSslCertificate`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/api/OnPremiseAuthenticatorService.kt:143)

#### Security Model

This feature is intended only for trusted on-premise environments. It should not be enabled casually in production because it disables normal certificate and hostname validation for the affected authenticator flows.

#### Example

```kotlin
class MyApplication : Application() {
    override fun onCreate() {
        super.onCreate()

        // Enable only if your trusted on-premise deployment requires it
        NetworkHelper.allowInsecureSSL = true
    }
}
```

### 2. QR Code Options Support for SSL Behavior

**Enhanced:** [`InitializationInfo`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/model/onprem/InitializationInfo.kt:23)

#### Overview

On-premise QR codes can now carry an `options` field that influences authenticator behavior. In 3.2.4, the relevant supported option is `ignoreSslCerts=true`.

#### Code Changes

- [`InitializationInfo`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/model/onprem/InitializationInfo.kt:23) derives `ignoreSSLCertificate` from the serialized `options` field
- The parsed value is propagated into on-premise authenticator creation and later used by [`MFAServiceController.initiate()`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/MFAServiceController.kt:129)

#### Example QR Payload

```json
{
  "details_url": "https://onprem.company.com/...",
  "code": "abc123",
  "client_id": "client-id",
  "options": "ignoreSslCerts=true"
}
```

## Bug Fixes

### 1. Correct Separation of Client and Server Authenticator Identifiers

**Fixed:** On-premise registration, refresh, and transaction flows now preserve the distinction between client-side `tenant_id` and server-side `authenticator_id`.

#### Problem

On-premise flows involve two different identifiers:

- **Client-side identifier**: generated by the SDK and used as the authenticator's local ID (`tenant_id`)
- **Server-side identifier**: returned by the server in token metadata (`authenticator_id`), often with a `uuid...` prefix

Earlier behavior and tests did not consistently enforce this distinction. That created risk in several areas:

- token refresh persistence could blur the two identifiers
- transaction filtering could use the wrong identifier
- registration finalization could incorrectly treat the server identifier as the local authenticator ID
- multi-authenticator scenarios could return or process transactions for the wrong account

#### Solution

The SDK now consistently treats these values as separate concerns:

- [`authenticator.id`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/api/OnPremiseAuthenticatorService.kt:137) remains the client-side `tenant_id`
- `token.additionalData["authenticator_id"]` remains the server-side identifier
- [`MFAServiceController.initiate()`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/MFAServiceController.kt:129) passes:
  - `_authenticatorId = authenticator.id`
  - `_serverAuthenticatorId = authenticator.token.additionalData["authenticator_id"]`
- token persistence callbacks continue to use the client-side authenticator ID
- transaction ownership filtering uses the server-side authenticator ID

#### Impact

- local persistence remains stable and keyed by the client-generated authenticator ID
- refreshed tokens preserve the server-side identifier in token metadata
- transaction retrieval and ownership checks align with server-side correlation rules
- multi-authenticator on-premise scenarios behave correctly

### 2. On-Premise Transaction Ownership Filtering

**Fixed:** [`OnPremiseAuthenticatorService.createPendingTransactions()`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/api/OnPremiseAuthenticatorService.kt:502) now correctly filters transactions using the server-side authenticator identifier.

#### Problem

The server associates pending transactions with the server-side `authenticator_id`, not the client-generated `tenant_id`. If filtering is performed using the wrong identifier, the SDK may:

- fail to return transactions that belong to the authenticator
- return empty results even when the server reports pending transactions
- incorrectly accept or reject transaction-specific lookups
- behave incorrectly in shared payloads containing transactions for multiple authenticators

#### Solution

[`OnPremiseAuthenticatorService.createPendingTransactions()`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/api/OnPremiseAuthenticatorService.kt:502) now:

- uses `_serverAuthenticatorId` as the ownership filter
- returns an empty transaction list when `_serverAuthenticatorId` is unavailable
- validates transaction-specific lookups against the server-side authenticator ownership
- preserves server-reported counts while returning only transactions owned by the current authenticator

#### Impact

- correct transaction isolation in multi-authenticator payloads
- correct behavior for transaction-by-ID lookups
- safer handling of incomplete or partially populated authenticator state
- improved parity with real on-premise server behavior observed in trace analysis

### 3. Registration and Token Refresh Persistence Semantics

**Fixed:** Registration finalization and token refresh flows now preserve the correct identifier semantics across persistence boundaries.

#### Problem

The SDK must preserve both:

- the client-side authenticator identity used for local storage and callbacks
- the server-side authenticator identity used for transaction correlation

If either is overwritten or conflated during registration finalization or token refresh, the authenticator can become inconsistent across subsequent operations.

#### Solution

The release strengthens the persistence contract across:

- [`OnPremiseRegistrationProvider`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/api/OnPremiseRegistrationProvider.kt)
- [`OnPremiseAuthenticatorService.refreshToken()`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/api/OnPremiseAuthenticatorService.kt:164)
- [`MFAServiceController.initiate()`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/MFAServiceController.kt:129)

The resulting behavior is:

- persistence callbacks receive the client-side authenticator ID
- refreshed tokens retain the server-side `authenticator_id`
- service creation reconstructs both identifiers correctly from persisted authenticator state

### 4. Registration Attribute Naming Consistency

**Fixed:** On-premise registration and refresh attributes continue to use the expected snake_case naming convention.

#### Overview

This release keeps the on-premise request model aligned with server expectations by ensuring fields such as account name and push token are represented consistently in snake_case.

#### Impact

- improved compatibility with on-premise server APIs
- reduced ambiguity between cloud and on-premise request conventions

## Improvements

### 1. Stronger Service Selection Logic for On-Premise Authenticators

[`MFAServiceController`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/MFAServiceController.kt:85) now more clearly centralizes the decision about which HTTP client should be used for on-premise service operations.

This improves maintainability by ensuring SSL-related client selection is handled in one place rather than being spread across multiple call sites.

### 2. Improved SDK Documentation for On-Premise Identifier Handling

Updated [`sdk/README.md`](../../sdk/README.md) to document the distinction between:

- local authenticator ID / `tenant_id`
- server-side `authenticator_id`

The documentation now explains which identifier should be used for:

- local persistence
- token refresh callbacks
- transaction filtering and correlation
- service construction

### 3. Expanded Regression Coverage Around Networking and Transaction Ownership

The release adds broader regression coverage around:

- secure vs insecure client selection
- client invalidation and recreation in [`NetworkHelper`](../../sdk/core/src/main/java/com/ibm/security/verifysdk/core/helper/NetworkHelper.kt:32)
- multi-transaction parsing and filtering
- cross-authenticator transaction rejection
- missing server identifier handling

### Migration Notes

No breaking API migration is required.

Applications using on-premise MFA should review the following:

1. If your environment uses self-signed certificates, explicitly enable [`NetworkHelper.allowInsecureSSL`](../../sdk/core/src/main/java/com/ibm/security/verifysdk/core/helper/NetworkHelper.kt:103) only when appropriate.
2. Ensure persisted on-premise authenticators preserve both:
   - local authenticator ID / `tenant_id`
   - token additional data containing server-side `authenticator_id`
3. If you construct services manually, ensure the correct identifier mapping is preserved as documented in [`sdk/README.md`](../../sdk/README.md).
