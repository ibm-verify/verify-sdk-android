# IBM Verify SDK for Android - Release 3.2.4

## Overview

Version 3.2.4 is a patch release focused on SSL certificate bypass support for on-premise authenticators with self-signed certificates, improved transaction filtering logic, and enhanced debugging capabilities. This release introduces a two-level security model for SSL bypass and fixes critical issues with authenticator ID handling in on-premise deployments.

## 🚨 Breaking Changes

None.

## ✨ New Features

### 1. SSL Certificate Bypass Support for On-Premise Authenticators

**New:** [`NetworkHelper.allowInsecureSSL`](../../sdk/core/src/main/java/com/ibm/security/verifysdk/core/helper/NetworkHelper.kt:103) property and [`NetworkHelper.createInsecureClient()`](../../sdk/core/src/main/java/com/ibm/security/verifysdk/core/helper/NetworkHelper.kt:278) method for controlled SSL bypass.

#### Overview

Enables applications to connect to on-premise IBM Verify Access servers with self-signed certificates through a two-level security model that requires both app-level permission and authenticator-level need.

#### Two-Level Security Model

1. **App-Level Permission** (`NetworkHelper.allowInsecureSSL`): Global flag that controls whether the application is allowed to bypass SSL validation at all. Must be explicitly enabled by the application developer.

2. **Authenticator-Level Need** (`ignoreSSLCertificate`): Indicates whether a specific authenticator requires SSL bypass (typically from QR code `"options":"ignoreSslCerts=true"`).

**Both conditions must be true** for SSL bypass to occur for a specific authenticator.

#### Usage Example

```kotlin
class MyApplication : Application() {
    override fun onCreate() {
        super.onCreate()

        // Enable SSL bypass capability for OnPrem authenticators
        // WARNING: Only enable if connecting to trusted servers with self-signed certificates
        NetworkHelper.allowInsecureSSL = true
    }
}
```

#### Security Warning

⚠️ **Enabling this in production exposes your application to man-in-the-middle attacks.** Only enable this if you need to connect to OnPremise servers with self-signed certificates that you trust.

#### Default Behavior

- **Default**: `false` (SSL bypass disabled - secure default)
- All authenticators use standard SSL certificate validation
- Attempting to use an authenticator with `ignoreSSLCertificate=true` will fail with an exception if `allowInsecureSSL` is `false`

#### When to Enable

Enable this flag only when:
- You need to connect to OnPremise IBM Verify Access servers
- Those servers use self-signed certificates
- You trust those servers and their network environment
- You understand the security implications

#### Implementation Details

- `NetworkHelper.createInsecureClient()` creates an HTTP client that trusts all certificates
- Throws `IllegalStateException` if called when `allowInsecureSSL` is `false`
- Used automatically during registration and service operations when `ignoreSSLCertificate` is `true`
- Comprehensive KDoc documentation with security warnings and usage examples

### 2. QR Code Options Parsing

**Enhanced:** [`InitializationInfo`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/model/onprem/InitializationInfo.kt) now parses `options` field from QR codes.

#### Overview

QR codes can now include an `options` field with configuration flags. Currently supports `ignoreSslCerts=true` for SSL bypass.

#### Implementation

```kotlin
@Serializable
internal data class InitializationInfo(
    @SerialName("details_url")
    val uri: URL,
    val code: String,
    @SerialName("client_id")
    val clientId: String,
    val options: String? = null
) {
    @Transient
    val ignoreSSLCertificate: Boolean = options?.contains("ignoreSslCerts=true") == true
}
```

#### QR Code Format

```json
{
  "details_url": "https://onprem.company.com/...",
  "code": "abc123",
  "client_id": "client-id",
  "options": "ignoreSslCerts=true"
}
```

## 🐛 Bug Fixes

### 1. On-Premise Authenticator ID Handling

**Fixed:** Critical issue where on-premise authenticators used server's `authenticator_id` instead of client-generated `tenant_id` for identification.

#### Problem

On-premise authenticators were storing the server's `authenticator_id` (returned in token response) as the authenticator's ID, but the server uses `tenant_id` for filtering transactions. This caused:
- Transactions not being found for the authenticator
- Mismatch between client ID and server filtering criteria
- Registration failures when server didn't return `authenticator_id`

#### Solution

Modified [`OnPremiseRegistrationProvider.initiate()`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/api/OnPremiseRegistrationProvider.kt:186) to:

1. Generate `tenant_id` on client side using `UUID.randomUUID()`
2. Send `tenant_id` to server during registration
3. Use `tenant_id` as the authenticator's ID (not server's `authenticator_id`)
4. Verify server returns `authenticator_id` but don't use it for client identification

```kotlin
// Generate tenant_id on client
val tenantId = UUID.randomUUID().toString()
attributes["tenant_id"] = tenantId

// Use tenant_id as authenticator ID
authenticatorId = tenantId
```

#### Impact

- Transactions now correctly filtered by `tenant_id`
- Authenticators work reliably across registration and transaction flows
- Consistent ID handling between client and server

### 2. Transaction Filtering Logic

**Fixed:** Transaction filtering in [`OnPremiseAuthenticatorService.createPendingTransactions()`](../../sdk/mfa/src/main/java/com/ibm/security/verifysdk/mfa/api/OnPremiseAuthenticatorService.kt:503) now uses server's `authenticator_id` for filtering.

#### Problem

Transaction filtering was using the client's authenticator ID (`_authenticatorId`) to filter transactions, but the server stores transactions with the `authenticator_id` it returned during registration. This caused transactions to not be found.

#### Solution

Added `_serverAuthenticatorId` field to store the server's `authenticator_id` from token response, and use it for transaction filtering:

```kotlin
// Use server's authenticator_id for filtering transactions
val filterAuthenticatorId = _serverAuthenticatorId

if (filterAuthenticatorId == null) {
    log.error("❌ No server authenticator_id available for filtering transactions")
    return emptyList()
}

// Filter transactions by server's authenticator_id
val identifiers = transactionResult.attributes.filter {
    it.uri == "mmfa:request:authenticator:id" && it.values.contains(filterAuthenticatorId)
}
```

#### Impact

- Transactions correctly filtered and retrieved
- No more "transaction does not belong to authenticator" errors
- Proper separation between client ID (tenant_id) and server ID (authenticator_id)

### 3. Registration Attribute Naming

**Fixed:** Attribute names in registration requests now use snake_case consistently.

#### Changes

- `accountName` → `account_name`
- `pushToken` → `push_token`
- Consistent with server API expectations

#### Impact

- Registration requests properly formatted
- Server correctly processes account name and push token
- Aligns with on-premise API conventions

#### Using SSL Bypass for On-Premise (New Feature)

If you need to connect to on-premise servers with self-signed certificates:

1. **Enable App-Level Permission**: Set `NetworkHelper.allowInsecureSSL = true` in your `Application.onCreate()`
2. **QR Code Configuration**: Ensure QR codes include `"options":"ignoreSslCerts=true"` if needed
3. **Security Review**: Understand the security implications before enabling in production

```kotlin
class MyApplication : Application() {
    override fun onCreate() {
        super.onCreate()
        
        // Only enable if you trust the on-premise servers
        if (BuildConfig.ALLOW_SELF_SIGNED_CERTS) {
            NetworkHelper.allowInsecureSSL = true
        }
    }
}
```

#### Authenticator ID Changes (Automatic Fix)

The authenticator ID handling fix is automatic and transparent:
- New registrations will use client-generated `tenant_id`
- Transaction filtering will use server's `authenticator_id`
- No code changes required in your application
